## PARTE 1 — IDENTIDADE

### 1.1 O que é a OTMA

A OTMA é uma consultora de processos operacionais para empresas que precisam implementar sistemas, ferramentas e IA — mas ficaram presas entre a intenção e a execução.

**Origem:** Processos desenvolvidos dentro da Unio por João e Symon. Testados em contexto real antes de virar produto.

**Princípio central:**
> "Não estamos ensinando do zero — estamos entregando a visão já estruturada."

**O que isso significa na prática:**
- Eliminamos a curva de aprendizado
- Traduzimos complexidade em passos simples
- Tornamos a execução possível mesmo para não especialistas

### 1.2 Brand Essence

> A maioria das empresas já sabe o que precisa fazer. O que falta é uma estrutura pronta para executar — sem precisar aprender do zero.

### 1.3 Propósito

> Consultoria de tecnologia ensina. Agências de automação vendem ferramentas. Nenhum dos dois entrega o que a empresa realmente precisa: um processo claro, nomeado, executável — que funciona mesmo sem especialista dentro de casa.
>
> A OTMA existe porque a maioria das soluções no mercado aumenta a dependência técnica do cliente, não reduz.

### 1.4 Narrativa Central

> A maioria das empresas não tem problema de intenção — tem problema de estrutura. Elas sabem que precisam organizar a operação, implementar ferramentas, começar a usar IA. Mas quando tentam, encontram consultores que ensinam, ferramentas que vendem e ninguém que simplesmente entrega um processo pronto para funcionar.
>
> A OTMA nasceu de dentro de uma operação real. Os processos que entregamos foram testados em contexto antes de virar produto. O que diferencia não é a ferramenta — é a forma de organizar o que precisa ser feito, de um jeito que qualquer pessoa do time consegue executar, e que uma IA consegue interpretar.
>
> O resultado não é uma apresentação. É um sistema operando.

## PARTE 2 — POSICIONAMENTO

### 2.1 Posicionamento

> A OTMA é para empresas que já têm a intenção de modernizar sua operação, mas ficaram presas no gap entre a decisão e a execução — porque as alternativas ensinam ou vendem tecnologia, mas ninguém entrega um sistema pronto para funcionar hoje.

**Alternativa real na cabeça do cliente:**
Não é outra agência. É a tentativa frustrada de fazer internamente — contratar alguém do time para "cuidar disso", ou assinar uma ferramenta que ficou sem uso.

### 2.2 Proposta de Valor

**Core Promise:**
> Quando você trabalha com a OTMA, você sai com um sistema operando — não com um manual de como operar.

**Razões para acreditar:**
- Os processos são entregues já nomeados, divididos em etapas e prontos para execução — sem necessidade de adaptação técnica pelo cliente
- A linguagem dos processos é legível por humanos e interpretável por IA — o mesmo sistema serve o time hoje e escala com automação depois
- A OTMA usa ferramentas disponíveis no mercado — sem dependência de plataforma proprietária ou lock-in
- Cada entrega é pensada para ser operada pelo cliente, não pela OTMA

### 2.3 Diferenciais

- **Processo > Ferramenta:** O valor não está no software escolhido — está na estrutura do pensamento por trás da implementação. Isso não é copiável comprando a mesma ferramenta.
- **Entrega, não ensino:** O cliente não precisa aprender a metodologia para se beneficiar dela. Isso filtra naturalmente quem quer consultoria e quem quer resultado.
- **Linguagem híbrida:** Os processos funcionam tanto para um colaborador não técnico quanto para uma IA que vai executar etapas automatizadas.
- **Origem prática:** Os processos foram testados dentro da Unio antes de se tornarem produto — não é framework teórico.

### 2.4 Público-Alvo

**Cliente primário:**
Empresas que já tentaram organizar operação, implementar alguma ferramenta ou "usar IA" — e não avançaram. Não por falta de orçamento ou intenção, mas porque ninguém entregou um sistema claro e pronto para rodar.

**Comportamentos que identificam o cliente:**
- Já perderam tempo com consultorias que ensinaram mas não implementaram
- Têm urgência de resultado, não disposição para virar especialistas
- Valorizam clareza acima de sofisticação técnica
- Estão dispostos a pagar por execução, não por teoria

**Quem NÃO é o cliente:**
- Empresas que querem aprender a fazer internamente (querem curso, não entrega)
- Gestores que precisam aprovar cada micro-decisão
- Quem busca tecnologia proprietária ou plataforma própria

## PARTE 3 — PROCESSOS OTMA

### 3.1 Lógica Operacional

**Input → Processamento → Saída**

| Etapa | Descrição |
|-------|-----------|
| **Entrada** | Problema ou necessidade da empresa |
| **Processamento** | Aplicação de um processo estruturado (etapas nomeadas) |
| **Saída** | Implementação prática de solução (ferramenta, sistema ou IA) |

### 3.2 Estrutura de Todo Processo OTMA

Todo processo segue este padrão de 4 camadas:

**1. Nomeação Estratégica**
- Criação de nomes próprios para cada etapa
- Linguagem simples e intuitiva
- Facilita memorização e execução

**2. Quebra em Etapas**
- Divisão clara do processo em partes menores
- Cada etapa tem uma função específica
- Organização sequencial e lógica

**3. Linguagem Híbrida**
- Compreensível por humanos
- Estruturada o suficiente para IA interpretar
- Evita termos técnicos desnecessários

**4. Foco em Implementação**
- Não é teórico
- Cada etapa leva a uma ação prática
- Orientado a execução

### 3.3 Lógica de Replicabilidade

Os processos são pensados para:
- Ser aplicados em diferentes empresas
- Adaptar-se a diferentes contextos
- Manter a mesma estrutura base

**Princípio:** Um mesmo modelo → múltiplas aplicações.

## PARTE 5 — TOM DE VOZ

**Em 4 adjetivos:** Direto. Específico. Sem overhead. Confiante sem precisar se explicar.

### Exemplos práticos

| Situação | OTMA diz | OTMA não diz |
|----------|----------|-------------|
| Descrevendo serviços | "Entregamos o processo estruturado e pronto para rodar." | "Transformamos sua empresa com soluções inovadoras." |
| Respondendo "quanto custa?" | "Depende do escopo — mas o ponto de entrada é X." | "Cada projeto é único, vamos conversar para entender sua realidade." |
| Falando de projeto finalizado | "A [empresa] saiu com [processo específico] funcionando em [prazo]." | "Foi incrível trabalhar com esse time incrível nessa jornada." |

**Regra de ouro:**
> A OTMA nunca publica conteúdo que ensina o cliente a fazer o que a OTMA faz. A autoridade vem de entregar, não de explicar.

## PARTE 7 — FERRAMENTAS INTERNAS

### 7.1 Skill Extractor (`/extract-skill`)

**O que faz:** Ao final de qualquer sessão de trabalho com um cliente, este comando analisa o histórico da conversa, extrai o padrão do processo executado, envia ao GPT-4 e retorna um arquivo `.skill` pronto para ser reutilizado em outros clientes.

**Fluxo:**
```
Sessão com cliente → /extract-skill → Claude analisa conversa
→ GPT gera SKILL.md estruturado → salvo em clients/otma/tools/skills/
```

**Etapas internas:**
1. **Mapear** — Identificar problema, etapas, inputs e saída da sessão
2. **Formatar** — Escrever PROCESS_SUMMARY estruturado
3. **Gerar** — Chamar GPT via `extract.py`
4. **Salvar** — Persistir em `clients/otma/tools/skills/[nome-do-processo]/SKILL.md`

**Requisito:** `OPENAI_API_KEY` definido no ambiente.

**Arquivos:**
- `clients/otma/tools/skill-extractor/SKILL.md` — instruções do skill
- `clients/otma/tools/skill-extractor/extract.py` — script Python que chama GPT-4o
- `~/.claude/skills/otma-skill-extractor.md` — instalação global (ativa o comando `/extract-skill`)

## PARTE 8 — ESTRUTURA DO PROJETO

### 8.1 Pastas

```
clients/otma/
├── CLAUDE.md              ← instruções para IA ao trabalhar neste projeto
├── OTMA-OS.md             ← este documento
├── brand/                 ← posicionamento, naming, tom, arquétipo, brand foundation
├── website/               ← sitemap, copy, estrutura, arquivos Framer
├── offers/                ← design de serviços, pacotes, precificação, propostas
├── operations/            ← processos, onboarding, fluxo de entrega
├── content/               ← conteúdo de autoridade, campanhas, editorial
├── assets/                ← logos, imagens, exports, materiais visuais
├── notes/                 ← rascunhos, pesquisa, material temporário
└── tools/
    ├── skill-extractor/   ← ferramenta de cristalização de processos
    └── skills/            ← skills geradas a partir de sessões com clientes
```

### 8.2 Regras de Operação

- Outputs vão em `clients/otma/` — não duplicar skills de referência
- Skills de metodologia ficam em `../../reference/` ou `~/.claude/skills/`
- Cada entrega para cliente deve resultar em pelo menos um skill salvo em `tools/skills/`
- Seções marcadas `[→ VALIDAR]` no brand-foundation.md precisam ser confirmadas pelos sócios antes de irem para o site

### 8.3 Workflow de Marca

1. `brand-dna` → fundações de identidade
2. `brand-os` → posicionamento, estratégia, lógica de ofertas
3. `scene-architect` → prompts visuais quando necessário
